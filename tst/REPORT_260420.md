## 3. Architecture and Design

### 3.1 Session Structure

```
Complex_Main (Isabelle/HOL standard library)
  └── Top0 (Chapters 2-8, general topology, 87K lines)
        ├── Top1_Ch2.thy (20.8K lines) — Topology foundations, products, subspaces
        ├── Top1_Ch3.thy (13.8K lines) — Connectedness, compactness
        ├── Top1_Ch4.thy (18.8K lines) — Countability, metric spaces
        └── Top1_Ch5_8.thy (33.6K lines) — Tychonoff, Urysohn, Baire, metrization
              └── AlgTop.thy (21.5K lines) — This formalization (§51-85)
```

### 3.2 Key Design Decisions

**Custom topology framework (`top1_*`):** The formalization uses a custom set-based topology framework (`top1_continuous_map_on`, `top1_connected_on`, `top1_compact_on`, etc.) rather than Isabelle's built-in type-class-based topology. This provides explicit carrier sets and topologies, matching Munkres' presentation, but requires bridges to HOL's `open`, `compact`, `connected` predicates.

**Unit interval representation:** `I_set = {0..1::real}`, `I_top = subspace_topology UNIV top1_open_sets I_set`, `II_topology = product_topology_on I_top I_top`. The product space I×I is central to homotopy theory.

**`by100` timeout wrapper:** All automation tactics are wrapped in `by (by100 tactic)` which enforces a 100ms timeout, preventing silent timeouts during builds. This was crucial for maintaining a ~33s build time.

### 3.3 Proof Engineering Patterns

Several patterns emerged as particularly effective:

1. **Sorry-first workflow:** Write the full proof structure with `sorry`, compile, then replace sorries in batches using `process_theories` + sledgehammer. This is mandated by the project's CLAUDE.md rules.

2. **Pasting lemma pattern:** Many continuity proofs decompose into: show two pieces are closed, show function is continuous on each piece, show they agree on overlap, apply `pasting_lemma_two_closed`.

3. **1D creeping lemma:** `open_cover_subdivision_01` provides subdivisions of [0,1] from pointwise open covers. Used for both path lifting (1D) and the Lebesgue grid (2D, applied twice).

4. **Monotonicity by induction on difference:** The pattern `proof (induction "b - a" arbitrary: b)` is used to prove monotonicity of increasing sequences (subdivision grids, boundary points).

5. **Hilbert choice (`SOME`) for function extraction:** When extracting functions from `∀x. ∃y. P x y`, we use `define f where "f x = (SOME y. P x y)"` followed by `someI_ex` to establish properties.

## 4. The FTA Proof in Detail

### 4.1 Proof Overview

The FTA proof follows Munkres §56 exactly:

**Step 1 (Polynomial reduction):** Reduce to showing that a monic polynomial `p(z) = z^n + a_{n-1}z^{n-1} + ... + a_0` with `n > 0` has a root.

**Step 2 (Loop degree computation):** For large enough `c > 0`, the restriction of `p` to the circle `|z| = c` is homotopic to `z ↦ z^n` (the lower-order terms become negligible). This gives a loop of winding number `n ≠ 0` in ℝ²-{0}.

**Step 3 (Nulhomotopic contradiction):** If `p` has no root, then `p` maps the closed disk `|z| ≤ c` into ℝ²-{0}. The restriction to the boundary circle would be nulhomotopic (extending to the disk). But Step 2 shows it has winding number `n ≠ 0`. Contradiction via π₁(S¹) ≅ ℤ.

### 4.2 Critical Dependencies

The FTA proof depends on a deep chain of results:

- **π₁(S¹) ≅ ℤ** (Theorem 54.5): requires the homotopy lifting property
- **Homotopy lifting** (Lemma 54.2): the most technically challenging formalization
- **Path lifting** (Lemma 54.1): the 1D precursor, itself non-trivial
- **Covering space theory** (§53): R → S¹ covering map, evenly covered neighborhoods

### 4.3 The Homotopy Lifting Proof (Lemma 54.2)

This was by far the longest and most difficult proof. The textbook proof is 1 page; the formalization is ~2000 lines. Key components:

1. **Edge lifts** (left edge, bottom edge via Lemma 54.1)
2. **Lebesgue grid existence** (Munkres: "Use the Lebesgue number lemma")
   - Pointwise ε-ball from continuity + covering map
   - 1D creeping lemma applied twice (t-direction, then s-direction)
   - Common refinement via `sorted_list_of_set`
   - Min-ε extraction, Archimedean N existence
3. **Rectangle-by-rectangle induction:**
   - A_k closed in I×I (finite union of closed sets)
   - Rectangle R closed (helper `closedin_II_rectangle`)
   - A_k ∩ R = L-shape (grid non-overlap argument)
   - L-shape connected (Theorem 23.3 + edge connectedness)
   - Homotopy lifting rectangle step (the key helper, ~300 lines)
4. **A(m×n) = I×I** (pigeonhole via `increasing_interval_cover`)
5. **Base case continuity** (pasting lemma on L-shape)
6. **Subspace topology self-equality**

## 5. What Went Wrong

### 5.1 The Lebesgue Grid Challenge

The textbook says simply "use the Lebesgue number lemma." Formalizing this required:

- `compact_Times` (product of compact sets is compact) is in HOL-Analysis but NOT in the `Complex_Main` import chain. We could not import it without risking conflicts.
- The `top1_lebesgue_number` from Top0 works with the `top1_*` framework, but bridging to the standard `open`/`compact` predicates required careful work.
- **The uniform grid approach (1/N intervals) does NOT work.** We discovered that intervals can straddle subdivision boundaries even when all pieces are wider than 1/N. The correct approach requires the actual common refinement (sorted union of boundary points).
- The final solution uses `sorted_list_of_set` and proves the refinement property via sorted list position arguments.

### 5.2 Topology Framework Bridge

The biggest recurring challenge was bridging between the `top1_*` framework (explicit carrier sets) and Isabelle's built-in topology (type-class based). Key bridge lemmas:

- `product_topology_on_open_sets`: product of `top1_open_sets` = `top1_open_sets` on product type
- `II_topology_eq_subspace`: II_topology = standard subspace topology on I×I
- `top1_continuous_map_on_II_to_I`: continuity transfer for I×I → I
- `top1_compact_on_subspace_UNIV_iff_compact`: bridge for compactness

Each bridge required careful type annotation and often by100 timeouts.

### 5.3 Automation Timeouts

The `by100` timeout wrapper (100ms) was both a blessing and a curse:
- **Blessing:** Prevented silent timeouts that would make builds take 2+ minutes
- **Curse:** Many "obvious" steps needed manual decomposition because blast/auto couldn't solve them in 100ms with the large proof context

Common patterns that caused timeouts:
- Set comprehensions with `top1_unit_interval_def` unfolding
- Product type membership extraction
- Complex `∃` instantiation from covering set definitions

### 5.4 Column vs. Rectangle Approach

Early attempts at Lemma 54.2 used a column-by-column approach (Hilbert choice per column) instead of the textbook's rectangle construction. This gave column-wise continuity but NOT joint continuity. The user had to repeatedly correct: "follow the textbook proof!" This lesson was eventually internalized but cost significant time.

### 5.5 Variable Name Collisions

In several proofs, Isabelle variable names collided with proof-level variables. For example, in `hpointball`, using `A` and `B` from `open_prod_elim` shadowed the covering map's base space `B`, causing type errors. These were subtle and time-consuming to debug.

## 6. What Went Well

### 6.1 The Sorry-First Workflow

The mandated sorry-first workflow was highly effective:
1. Write full proof structure with sorry → build → verify structure
2. Replace sorries in batches → build → verify
3. If anything breaks, revert to last working state

This prevented the common failure mode of writing a long proof that doesn't compile.

### 6.2 Helper Lemma Extraction

Extracting reusable helper lemmas was crucial:
- `closedin_I_sub_interval`: closed sub-intervals of I
- `closedin_II_rectangle`: closed rectangles in II_topology
- `increasing_interval_cover`: pigeonhole for monotone subdivisions
- `grid_from_per_piece_subdivisions`: common refinement helper

These made the main proofs much cleaner.

### 6.3 Textbook Fidelity

Once we committed to following the textbook proof exactly (rectangle-by-rectangle construction for Lemma 54.2), progress was much faster. Deviations from the textbook consistently led to dead ends.

### 6.4 Incremental Building

The ~33s build time made rapid iteration possible. Each sorry replacement could be tested immediately. The numbered backup system (bck0001, bck0002, ...) provided safety nets.

### 6.5 The Common Refinement Breakthrough

The final breakthrough was realizing that:
1. The uniform grid approach (1/N) doesn't work (intervals straddle boundaries)
2. The correct approach uses `sorted_list_of_set` on the union of all boundary points
3. The refinement property follows from: consecutive elements in the sorted list have no subdivision boundary strictly between them

This was proved via a position argument: if a boundary point exists at position m in the sorted list, then m > k (from sorted order), so m ≥ k+1, giving the containment.

## 10. Lessons Learned

1. **Follow the textbook.** Every deviation from the textbook proof led to a dead end. The textbook's proof structure is optimized for the logical dependencies.

2. **Sorry-first is essential.** Writing proof structures before filling in details prevents wasted effort on proofs that don't fit the overall structure.

3. **Build often.** The ~33s build time enabled rapid iteration. Longer build times would have been crippling.

4. **Bridges are expensive.** The largest time investment was bridging between the custom `top1_*` framework and Isabelle's standard library. Future projects should consider using HOL-Analysis directly.

5. **The "one sentence" in the textbook can be the hardest step.** Munkres' "Use the Lebesgue number lemma" required ~500 lines of formal proof including the common refinement construction.

6. **Isabelle's `sorted_list_of_set` is useful.** For constructing increasing sequences from finite sets, this avoids manual sorting proofs.

7. **`by100` timeout is a good practice.** It surfaces slow proofs immediately rather than letting them accumulate into session timeouts.

A sorry statement was discovered in `Theorem_56_1_step_1` (line 14049) that was
missed by the earlier grep pattern `^\s*sorry\b`. The sorry is at the END of a
line: `... f g" sorry`, not at the start.

**FTA dependency chain:** FTA → step_3 → step_2 → step_1 → sorry (hinj)

The sorry states: if `(f(s))^n` is path-homotopic to `(g(s))^n` on S¹, then
`f` is path-homotopic to `g`. This is the injectivity of the z^n-induced map
on π₁(S¹).

**Proof strategy:** Under the isomorphism π₁(S¹) ≅ ℤ (Theorem 54.5), the z^n
map corresponds to multiplication by n. If n·k = n·l and n > 0, then k = l.
This requires showing that the standard loop composition with z^n lifts to
multiplication by n in the covering space ℝ.
